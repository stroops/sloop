# Sloop MVP — Design Spec

**Date:** 2026-06-26
**Status:** Approved (design), pending implementation plan
**Scope:** v1 (MVP). Items marked _(v2)_ are explicitly out of scope.

---

## 1. Positioning

Sloop is a **lightweight, local-first developer-experience layer that sits _above_ existing
AI coding CLIs** (Claude Code, Cursor CLI, Aider, future local agents). It does **not**
replace or proxy those tools.

Its job is narrow and concrete:

1. Keep a **canonical context/skills/vault** for a project in `.sloop/`.
2. **Sync** that canonical content into each tool's **native context file** (`CLAUDE.md`,
   `.cursor/rules`, `AGENTS.md`, …).
3. **Launch** the chosen tool in the right working directory, optionally multiplexed with
   tmux, and remember what you launched.

There is **no daemon, no gRPC, no vector DB, no API/LLM routing**. Everything is a
run-to-completion CLI command reading/writing files and a local SQLite DB. This resolves
the contradiction in the original README/OVERVIEW, which promised "lightweight, local-first"
while diagramming a daemon + gRPC + embeddings + LiteLLM router.

### MVP anchor

The first version targets the **solo developer**: a workspace + launcher core. The other
use cases discussed (2nd brain, reusable skills, non-CLI team members) layer on top of the
same core. The **non-dev guided UX (BA/QC)** is explicitly a later surface _(v2)_.

---

## 2. Architecture

### 2.1 Single binary

One binary: `sloop`. The daemon scaffolding is removed.

**Cleanup of current scaffolding:**
- Delete `cmd/sloopd/` and the daemon/socket concept.
- Remove the `--socket` persistent flag from `internal/cli/root.go`.
- Fix the broken `internal/store/sqlite.go` (does not compile) and move it to
  `internal/session`.
- Keep the existing Cobra auto-register command pattern.

### 2.2 Package layout

```
cmd/sloop/            # entrypoint (kept)
internal/
  cli/                # cobra commands (keep auto-register pattern)
  config/             # global + project config (YAML)
  workspace/          # resolve & read .sloop/; walk-up + registry lookup
  adapter/            # load YAML manifests; render outputs; expose launch cmd
  sync/               # assemble canonical context -> render -> write native files
  profile/            # named launch presets within a workspace
  session/            # SQLite store: workspaces registry + session history
  detect/             # on-demand detection of adapters/tools + tmux
  runner/             # launch: tmux-aware, with plain-exec fallback
  doctor/             # environment health report (reuses detect)
  status/             # `sloop status` line
```

Design principle: each package has one purpose and a small surface, testable in isolation.

## 3. Workspace & working-directory model

A **workspace = a directory containing `.sloop/`** (source of truth, git-friendly). A
workspace enables multiple tools; every tool is launched with **cwd = the workspace root**
(where `.sloop/` lives). This matches how the tools actually work: cd into a folder, then
run e.g. `claude`.

**Two-layer targeting:**

1. **Directory-based (source of truth):** when inside a project, Sloop walks up from cwd to
   find `.sloop/` (like git finding `.git/`). That directory is the workspace root.
2. **Global registry (convenience):** `sloop init` registers the workspace in
   `~/.sloop/sloop.db` (name → absolute path). From anywhere, `sloop run -w <name> <tool>`
   resolves the path, cd's there, and launches. This is what addresses the "too many
   windows/sessions" pain — no manual cd, no remembering paths.

**Workspace ↔ tools ↔ profile:**

```
Workspace = one folder + .sloop/
  ├─ enabled tools: [claude, cursor, aider]   # each launches at this folder
  ├─ context/ , skills/ , vault/
  └─ profiles/ (optional named presets): preset = one tool + a selected set of skills/context
       e.g. "claude-review"  = claude + review skills
            "aider-refactor" = aider  + refactor skills
```

`sloop run` accepts a **tool** (`sloop run claude`) or a **profile**
(`sloop run claude-review`), with optional `-w <workspace>` to target another workspace
without cd'ing.

**Profile file shape** (`profiles/<name>.yaml`):

```yaml
tool: claude          # which adapter to launch (required)
context: all          # "all" (default) or an explicit list of context/*.md filenames
skills: []            # explicit list of skills/*.md to include (default none)
vault: []             # explicit list of vault/*.md to include (default none)
```

The default profile generated by `init` for a detected tool is `{ tool: <tool>, context: all,
skills: [], vault: [] }`. Running a bare tool name (`sloop run claude`) uses that default
profile if present.

## 4. Storage layering

| Location | Contents | Git? |
|---|---|---|
| `<project>/.sloop/` | `config.yaml`, `context/`, `skills/`, `vault/`, `profiles/` | ✅ committed (shared with team) |
| `<project>/.sloop/.gitignore` | ignores generated native files + caches | ✅ committed |
| generated native files (`CLAUDE.md`, `.cursor/rules`, `AGENTS.md`) | rendered output at workspace root | ❌ gitignored |
| `~/.sloop/config.yaml` | global config (default mode, defaults) | ❌ machine-local |
| `~/.sloop/adapters/*.yaml` | user-added tool manifests | ❌ machine-local |
| `~/.sloop/sloop.db` | SQLite: workspaces registry + session history | ❌ machine-local |
| `~/.sloop/vault/` _(optional)_ | personal global vault, available to all projects | ❌ machine-local |

**Rationale:** the registry and history are machine-local and personal (absolute paths
differ per machine; history is yours), so they live in `~/.sloop/`, never in the project's
`.sloop/`. The project `.sloop/` holds only shareable, committable workspace definitions.

### `.sloop/` (per-project) layout

```
<project>/.sloop/
  config.yaml         # enabled tools, default tool, per-project mode override
  context/            # canonical context (markdown) — source of truth
    project.md
  skills/             # *.md reusable snippets
  vault/              # *.md 2nd-brain notes (referenced/included into context)
  profiles/           # *.yaml launch presets
  .gitignore
```

## 5. Context delivery (sync)

Canonical content lives in `context/` + `skills/` + `vault/`. Native files are a
**generated product** (gitignored).

**`sloop run <tool>` flow:**

1. **Resolve workspace** — cwd walk-up, or `-w <name>` registry lookup → root dir + `.sloop/`.
2. **Resolve target** — a tool, or a profile (→ tool + selected skills/context subset).
3. **Assemble canonical context** — concatenated in a fixed order:
   1. all `context/*.md` (alphabetical by filename),
   2. the profile's selected `skills/*.md` (in the order listed in the profile),
   3. the profile's selected `vault/*.md` (in the order listed in the profile).
   Each source file is separated by a `## <filename>` heading so the origin is traceable in
   the generated output.
4. **Render & write native files** — for the tool's adapter manifest, render each
   `output` template with the assembled context, writing native files (e.g. `CLAUDE.md`)
   at the workspace root.
5. **Record session** — write a row to SQLite (best-effort; a write failure warns but does
   not block the launch).
6. **Launch** — tmux-aware (see §8), else plain exec with `cwd = root`.

`sloop sync [tool|--all]` runs steps 1–4 only (no launch), to regenerate native files.

## 6. Tool adapter model

Adapters are **declarative YAML manifests** — data, not code. Adding a tool is adding a
YAML file, not editing Go. Built-in manifests are embedded via `go:embed`; users add their
own at `~/.sloop/adapters/*.yaml`.

```yaml
# internal/adapter/builtin/claude.yaml
name: Claude Code
detect: claude            # binary name checked on PATH
launch: claude            # launch command
outputs:
  - path: CLAUDE.md       # native file written at workspace root
    template: default     # body = assembled context
```

**v1 manifest fields:** `name`, `detect`, `launch`, `outputs[].path`, `outputs[].template`.

**v1 built-in adapters:** Claude Code, Cursor CLI, Aider (the architecture is pluggable; if
time-constrained, Claude Code alone is acceptable for the first cut, with the others added
as data).

_(v2)_ `install_hint` field (per-OS install suggestions). Detection in v1 reports
installed/missing without an install hint.

## 7. Sessions & SQLite

DB file: `~/.sloop/sloop.db`. Driver: `modernc.org/sqlite` (pure-Go, no cgo → runs on
macOS/Linux/Windows). Migrations are embedded SQL executed on store open.

```sql
CREATE TABLE workspaces (
  id         INTEGER PRIMARY KEY,
  name       TEXT UNIQUE,
  path       TEXT UNIQUE,
  created_at TIMESTAMP
);

CREATE TABLE sessions (
  id           INTEGER PRIMARY KEY,
  workspace_id INTEGER REFERENCES workspaces(id),
  tool         TEXT,
  profile      TEXT,
  cwd          TEXT,
  tmux_session TEXT,
  started_at   TIMESTAMP,
  ended_at     TIMESTAMP NULL
);
```

Session writes are best-effort: failures warn and never block a launch.

## 8. tmux integration (optional)

tmux is an **optional enhancement, never a hard dependency** — Windows lacks native tmux,
and the future BA/QC users won't have it.

- **Detect:** `exec.LookPath("tmux")`.
- **With tmux:** session named `<workspace>__<tool>` (sanitized — no `.`/`:`). If the
  session exists → attach; otherwise `tmux new-session -c <root> -s <name> '<launch>'`.
  `sloop attach <name>` → `tmux attach -t <name>`.
- **Without tmux:** fall back to `exec.Cmd{Dir: root}` running the launch command in the
  current terminal; log an info line only.

Chosen over Zellij deliberately: tmux is lighter, ubiquitous, and its CLI
(`new-session`/`new-window`/`send-keys`/`list-sessions`/`attach`) is all we need.

## 9. Detection & auto-enable

### 9.1 Detection (on-demand, no daemon, no cache)

One `detect` component scans:
- **Each registered adapter** (built-in + user) → `exec.LookPath(detect)` → installed/missing,
  plus best-effort version (`<bin> --version` with a timeout; failure ignored).
- **tmux** → `exec.LookPath("tmux")` + `tmux -V`.

Reused in three places (no duplicated logic):
1. `sloop doctor` — full health report.
2. `sloop init` — auto-enable detected tools (see §9.2).
3. `sloop run` — pre-flight: missing target tool → clear error; missing tmux → fallback.

**Limitation (explicit):** detection only knows tools that have a manifest. To detect a new
tool, add a YAML manifest — this is the upside of the declarative adapter model.

### 9.2 Auto-enable (v1 scope)

Controlled by interaction **mode** (§10). v1 includes:

- **`init` generates ready-to-run profiles:** scan installed tools → for each found tool,
  enable it in `config.yaml` **and** create a default `profiles/<tool>.yaml` (tool + full
  context, no skill filter). After `init`, `sloop run <tool>` works with zero extra config.
- **`run` smart fallback:** no `.sloop/` in cwd and no `-w` → in `ask` mode, prompt "init
  here? (y/N)" instead of a hard error; no target given → use the workspace's default tool.

_(v2)_ `init` auto-running `sync`; `install_hint` output for missing tools.

## 10. Interaction mode (the simplicity knob)

A single setting `mode` in `~/.sloop/config.yaml` (overridable per-project and by flag),
instead of hardcoding aggressive behavior:

- **`ask` (default):** prompt before doing convenient-but-intrusive things (init-on-the-fly,
  default-tool selection). Safe, predictable.
- **`auto`:** do them silently.
- **Flags:** `--auto`/`-y` forces auto for one invocation; `--no-input` for scripts/CI
  (fail instead of blocking on a prompt).

**Safety boundary (hard rule):** Sloop **never installs software or runs a package manager**
(npm/brew/curl). Auto-enable only ever creates config/profile files inside the project. This
is non-negotiable regardless of mode.

## 11. `sloop status` (Sloop's own statusline)

A command printing one compact line of **Sloop-level** context:

```
⚓ backend · claude · tmux:backend__claude · sync:fresh
```

(workspace · active/last tool · tmux session · whether native files are fresh vs stale)

`sync:fresh` vs `sync:stale` is determined by comparing modification times: native files are
**stale** if any canonical source (`context/`, the active profile's selected `skills/`/`vault/`,
or the profile file itself) has an mtime newer than the generated native file. Otherwise
**fresh**.

This is **Sloop's own outer-level statusline** — distinct from the AI provider's statusline
that runs _inside_ each tmux window (e.g. Claude Code's own status line). Sloop does not
touch the provider's statusline; it gives the orchestration-level view.

v1 scope: the `sloop status` command only (plain output; `--format json` optional). Deep
integration into tmux's status bar, the shell prompt, or Claude Code's `statusLine` setting
is _(v2)_.

## 12. Command set (v1)

```
sloop init [--name <n>]                  # create .sloop/ + register workspace; auto-enable detected tools + profiles
sloop sync [tool | --all]                # regenerate native files from canonical context
sloop run [tool|profile] [-w <ws>] [-- <args>]   # sync + cd + launch (tmux-aware)
sloop ls                                 # list workspaces + recent sessions
sloop attach <name>                      # attach a tmux session (if tmux present)
sloop tools                              # list adapters + installed status
sloop doctor                             # environment health (tools, tmux, config)
sloop status                             # Sloop statusline (one line)
sloop skill new <name>                   # convenience: scaffold a skill file (skills/vault are just .md)
```

Global flags: `--config`, `--no-color`, `--auto`/`-y`, `--no-input`. (The `--socket` flag is
removed.)

---

## 13. Error handling

- No `.sloop/` found and no `-w` → in `ask` mode, prompt to `init`; in `--no-input`, error
  pointing at `sloop init`.
- Target tool not installed → clear error, suggest `sloop doctor`.
- tmux not installed → silent fallback to exec (info log only).
- SQLite write failure → warn, continue (history is best-effort).

---

## 14. Testing (TDD)

- **Unit:** workspace resolution (walk-up + registry lookup); context assembly ordering;
  manifest rendering (golden files); session store (sqlite in-memory).
- **Runner isolation:** the launch step sits behind an interface so tests assert the
  *constructed command* (exec args / tmux args) without actually launching a process.
- **Integration:** `init` + `sync` produce the expected native files in a temp dir;
  detection against a faked PATH.

---

## 15. Out of scope (v2+)

- Daemon, gRPC, unix-socket IPC.
- Vector DB / embeddings; LLM/API routing (LiteLLM); Sloop acting as a chat client.
- `install_hint` / any auto-install of tools.
- `init` auto-sync.
- Deep statusline integration (tmux status bar, shell PS1, Claude Code `statusLine`).
- Non-developer guided UX for BA/QC.
- tmux as a required dependency.
